## 1. Стратегія тестування

Для забезпечення надійності системи було впроваджено трирівневу піраміду тестування:

1.  **Unit-тести (Jest):** Повне покриття бізнес-логіки модуля бронювання (`booking.service.ts`).
2.  **Інтеграційні тести (Supertest):** Перевірка взаємодії маршрутів, контролерів та сервісів.
3.  **E2E-тести (Playwright):** Перевірка повного клієнтського сценарію в реальному браузері (Авторизація -> Вибір номера -> Бронювання -> Скасування).

## 2. Unit-тестування модуля бронювання

Об'єктом тестування обрано `booking.service.ts`, оскільки він містить найкритичнішу логіку:

- Розрахунок вартості проживання.
- Перевірка перетину дат з існуючими бронюваннями.
- Перевірка доступності номера з урахуванням журналу технічних робіт.

**Результат:** Тести успішно виявляють помилки в логіці "Overlap" та некоректних вхідних датах. Використано `jest.spyOn` для ізоляції від бази даних Prisma.

## 3. Е2Е тестування

Реалізовано сценарій автоматизації в браузері Chromium:

1.  Вхід під існуючим користувачем.
2.  Перехід на сторінку категорії номера.
3.  Вибір конкретної кімнати.
4.  Вибір періоду в майбутньому (жовтень 2026 рік) через перемикання місяців у календарі.
5.  Створення бронювання.
6.  **Cleanup (Очищення):** Автоматичне скасування броні для підтримання ідемпотентності тестів.

## 4. Звіт про мутаційне тестування (Stryker)

### Результати в цифрах:

- **Загальний Mutation Score:** 4.57%
- **Mutation Score для `booking.service.ts`:** 30.51%
- **Mutation Score для `booking.controller.ts`:** 13.04%
- **Вбито мутантів (# killed):** 41
- **Вижило мутантів (# survived):** 1254

### Аналіз ефективності покриття:

1.  **Низький загальний бал (4.57%):** Пояснюється тим, що згідно з вимогами ЛБ №5, глибоке покриття Unit-тестами було реалізовано лише для одного модуля. Більшість інших файлів (Auth, Payment, Room Services) мають 0%, оскільки покриті лише базовими інтеграційними тестами, які не перевіряють граничні значення.
2.  **Аналіз модуля бронювання (30.51%):** Це найкращий показник у системі. Тести ефективно вбивають мутантів, що змінюють арифметичні оператори та логіку порівняння дат.
3.  **Чому мутанти вижили:**
    - **Неповні твердження (Assertions):** Тести перевіряють факт успіху операції, але не завжди перевіряють повну структуру повернутого об'єкта.
    - **Обробка помилок:** Мутації у блоках `catch` та повідомленнях про помилки часто залишаються непоміченими.
    - **Необов'язкові поля:** Зміни в обробці `specialRequests` не призводять до падіння тестів.

## 5. Висновки та рекомендації

Поточне тестування довело працездатність основних бізнес-сценаріїв готелю. Проте, мутаційний аналіз виявив "сліпі зони" в архітектурі тестів.

**Що варто зробити для покращення якості:**

1.  **Розширення Unit-покриття:** Додати тести для `auth.service.ts` (логіка JWT) та `payment.service.ts` (обробка LiqPay).
2.  **Поглиблення перевірок:** У тестах бронювання використовувати `expect.objectContaining` для детальної перевірки кожного поля, що записується в БД.
3.  **Тестування валідації:** Додати окремі тести для схем Zod, щоб переконатися, що жодна некоректна мутація в правилах вводу не виживе.
